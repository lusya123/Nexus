# OpenClaw Nexus —— AI Agent 军团监控与控制中心

> 说明（更新于 2026-02-16）：本文档是产品愿景，不对应当前版本功能清单。当前已实现能力请查看 `README.md` 与 `docs/`。

## 产品愿景文档 v1.0

---

## 一、核心愿景

OpenClaw Nexus 是一个三维可视化的 AI Agent 军团监控与控制中心。它的核心隐喻是**一栋大楼**——用户站在大楼外面可以看到全局运行状态，走进大楼可以看到各个部门，走进部门可以看到具体任务和 Agent。

这个系统要解决的根本问题是：当用户同时运行着数以万计的 AI Agent（包括 OpenClaw Session、Claude Code、Codex 等），如何让用户像一个老板管理一家公司一样，实时监控和指挥这些"无限心智"。

整个界面是一个真正的 3D 场景，用户可以旋转、拖拽、zoom in/zoom out，在不同层级之间自由穿梭。

---

## 二、信息架构：四维模型

### 2.1 四个维度

系统需要承载四个信息维度：

**维度一：部门（深度轴）。** 不同的业务功能单元，每个部门处理一类任务。例如研发部、内容部、数据采集部、客服部等。在 3D 场景中表现为大楼的不同楼层，或前后堆叠的扁平长方体。

**维度二：任务状态（横轴）。** 任务在流程中的当前阶段。全局统一采用三列模型：待办 → 执行中 → 已完成。在看板中表现为从左到右的三个区域。选择三列而非五列的原因是：在 Agent 的世界里，Planning、Testing、Review 这些中间状态通常由同一个 Agent 在一次连续执行中完成，不像人类团队需要在不同阶段交接。如果某些部门确实需要更细的状态，可以在部门内部自行扩展，但全局视图保持三列。

**维度三：任务 ID（纵轴）。** 同一部门、同一状态列内的具体任务，纵向排列。每个任务是一个小的卡片/长方体。

**维度四：Agent（内部轴）。** 每个任务内部正在执行的智能体。每个任务卡片里面包含一个或多个 Agent，每个 Agent 对应一个实时滚动的终端窗口。Agent 的类型包括 OpenClaw Session、Claude Code Session、Codex Session，以及任何有智能体介入运作的 Session。

### 2.2 层级包含关系

这四个维度并非等价的坐标轴，而是存在天然的包含关系：

**集群/机器 → 部门 → 任务（带状态）→ Agent/Session**

这是一棵树。用户不需要同时看到所有维度的全部信息。通过缩放和下钻，用户在不同层级上看到的信息量始终被控制在人脑可以舒适处理的范围内。

### 2.3 关于部门的差异化

不同部门的 UI 结构统一——都是三列看板加 Agent 终端。不需要模板系统，不需要用户配置。不同部门"感觉不同"的关键在于两点：一是数据不同（里面跑的任务和 Agent 不同），二是置顶指标不同（客服部置顶"响应时间"和"满意度"，内容部置顶"发布量"和"播放量"）。置顶指标可以由部门的 Leader Agent 自动设定，也可以由用户手动选择。

## 三、3D 交互模型：大楼隐喻

### 3.1 缩放层级与视角过渡

整个交互体验类似 Google Earth——从太空缩放到街景的过程中，用户感觉自己在一个 3D 空间里移动，但真正阅读信息和操作的时候，界面自然过渡到了适合阅读的形态。

#### 最远距离：集群视图

用户看到一栋或多栋 3D 建筑。每栋建筑代表一台机器或一个集群。建筑表面有微光流动，表示内部有 Agent 在运行。建筑顶部悬浮显示关键数字：总 Agent 数、任务完成率、收入等。

当用户新增一台云服务器时，界面上丝滑地浮现一栋新建筑，与现有建筑并列排布。建筑之间有光线/粒子连接，代表跨机器的任务调度和通信。

用户可以自由旋转和拖拽这些建筑，从不同角度观察整个集群的运行状态。

#### 中等距离：部门视图

用户推近某栋楼，楼体变成半透明，内部一层一层的楼层显现出来。每层是一个部门，侧面有部门名称和状态指示灯（绿色正常、红色异常）。楼层之间有光线在跑，代表任务在部门间流转——例如数据采集部完成的数据会有一条闪光线跟随着一个任务卡片飞到数据分析部的待办列。

#### 近距离：看板视图

用户再推近某一层楼。视角自然过渡：3D 楼层"展开"成一个 2D 的看板界面。动画的感觉是从楼外面飞进了办公室内部，墙壁往两边退开，桌子上的东西铺在面前。

内部是标准的三列看板（待办 / 执行中 / 已完成），每列下纵向排列着任务卡片。任务完成时，卡片自动向右滑动到下一列。

#### 最近距离：Agent 终端视图

用户点击某个任务卡片，卡片展开，内部显示一个或多个模拟终端窗口。每个终端对应一个正在工作的 Agent/Session，里面实时滚动着命令行输出——字符不断滚动，就像黑客电影里的监控室画面。

终端窗口上方标注 Agent 类型（OpenClaw Session / Claude Code / Codex）、运行时长、Token 消耗。

后期目标：用户可以点进某个终端窗口，直接输入指令与该 Agent 交互。

### 3.2 导航操作

用户可以在 3D 场景中进行以下操作：

- **旋转**：拖拽旋转整个场景或某栋建筑，从不同角度观察
- **缩放**：鼠标滚轮或双指缩放，在不同层级之间平滑过渡
- **点击下钻**：点击某栋楼进入部门视图，点击某层进入看板视图，点击某个任务进入终端视图
- **返回上钻**：按 Escape 或双指缩小，动画反向播放，逐级退出

### 3.3 手势控制（远期）

- **手掌张开**：大楼瞬间"爆炸"成平铺视图，所有 Agent 终端窗口平铺在屏幕上，就像黑客电影里的监控室——几十个终端同时滚动输出。
- **手指收拢**：终端窗口飞回各自的格子，重新组装成大楼形态。就像把员工放回工位一样。

## 四、数据流与动画

### 4.1 任务流转动画

当一个任务完成并产生下游任务时，界面上会出现可视的流转动画：

- **看板视图内**：任务卡片从"执行中"列平滑滑动到"已完成"列
- **部门视图内**：楼层间有闪光线/箭头跟随着任务卡片从一个部门飞到另一个部门的待办列
- **集群视图内**：跨机器的任务调度表现为建筑之间的粒子连线

整个大楼内部有大量这样的任务在不停流转，每一个任务又有里面很多的 Agent 在执行，形成一个繁忙而有序的视觉场景。

### 4.2 Agent 活动指示

每个任务卡片即使在未展开状态，也能看到内部 Agent 的活动指示：

- 微型字符滚动的动画覆盖在卡片表面，表示内部有 Agent 在工作
- 卡片边缘的颜色/亮度反映 Agent 数量和活跃度
- 异常状态的卡片会变红/闪烁，自动吸引注意力

### 4.3 输入流：对话即任务

用户与 OpenClaw 的正常对话就是创建任务的入口。用户说一句话，就会看见一个任务卡片被抛入某个部门的待办列，然后 Agent 自动接手开始执行。前端就像饭店的入餐口——源源不断有任务进来，被分配到各个部门的流水线上。

两类交互方式共存：第一是人调用 AI（用户通过对话创建任务），第二是 AI 调用 AI（一个 Agent 可能调用几万个子 Agent 同时处理任务）。

## 五、指标体系

### 5.1 全局指标（老板视角）

在集群视图的顶部或建筑上方悬浮显示，让老板一眼知道业务和 Agent 公司运行得怎么样：

- **业务指标**：今日流量、涨粉数、点赞数、收入、利润
- **运行指标**：当前运行 Agent 数、任务完成速率、今日 Token 消耗
- **健康指标**：异常任务数、失败率、平均响应时间

具体放哪些数字可以根据业务需要调整，但一定是能反映业务状态的关键数字。

### 5.2 部门级指标

每个部门有不同的置顶指标，这是部门之间"感觉不同"的关键：

- **内容部**：今日发布量、播放量、平均互动率
- **客服部**：平均响应时间、客户满意度、处理工单数
- **数据部**：数据采集量、处理吞吐量、异常数据比例
- **研发部**：代码提交数、Bug 修复率、部署频率

### 5.3 Agent 级指标

每个 Agent 头上悬浮显示其核心指标：

- **ROI 比率**：消耗的 Token 与产出的收益的比率
- **北极星指标**：由部门的领导 Agent 根据业务目标自动设定的核心衡量指标

## 六、专用视图（远期）

除了核心的大楼视图和看板视图外，特定业务场景可能需要专用视图。这些视图作为可选的视角切换，而非默认界面。

### 6.1 客服视图

以对话流为中心的展示。显示每个客服 Agent 当前正在处理的对话，包括国内社群运营和海外社群运营的 Agent 如何回复。每个客服 Agent 上方悬浮显示响应时间和满意度指标。能够展现出这么多 Agent 同时运营的场景，以及衡量他们的关键数据。

### 6.2 内容生产视图

展示内容引擎的完整流水线：

- **数据输入层**（顶部）：同行监控、热点抓取、历史作品数据
- **内容生产层**（中间）：内容生产的任务看板，内部细分为不同的内容类型组，每组里有多个 Agent 执行任务
- **产出与数据反馈层**（底部）：已发布内容的实时数据表现

### 6.3 实验视图

以树状结构展示正在进行的 A/B 测试和实验。例如：

- 测试不同的钩子（Hook）：1号钩子、2号钩子……
- 测试不同的选题：1号选题、2号选题……
- 每个分支显示真实世界的数据反馈：抖音播放量、小红书点赞率、转化率

这个视图是产生自我优化的关键——Agent 必须能主动做实验、获取真实数据、迭代优化。实验视图可应用于所有部门，包括客服、自我迭代等。

## 七、多机器与多集群

### 7.1 单机到多机

每台机器（本地或云服务器）在 3D 场景中表现为一栋独立的建筑。多台机器并列排布。新增一台云服务器时，界面上丝滑地浮现一栋新建筑。

机器之间有跨机器的任务调度和通信，表现为建筑间的光线和粒子效果。任务在机器间跳跃、流动。

### 7.2 集群合并与展开

多台机器可以逻辑上合并为一个更大的集群：

- **合并动画**：多栋建筑平滑地组合成一个更大的结构体/矩阵
- **并列模式**：也可以保持并列状态，中间有任务到处发送、到处跳跃、不断流动

在总页面（合并后的集群视图）上展示最关键的聚合指标。

## 八、核心数据模型

无论前端如何展示，后端需要稳定的数据模型。五个核心实体：

**Machine（机器/集群）**：标识一台物理机器或逻辑集群。包含机器名称、地址、状态、包含的部门列表。

**Department（部门）**：标识一个业务功能单元。包含部门名称、所属机器、置顶指标配置、包含的任务列表。

**Task（任务）**：标识一个具体的工作任务。包含任务名称、所属部门、当前状态（待办/执行中/已完成）、创建时间、完成时间、分配的 Agent 列表。

**Session/Agent（执行单元）**：标识一个正在工作的智能体。包含 Agent 类型（OpenClaw / Claude Code / Codex）、所属任务、实时输出流、运行时长、Token 消耗。

**Metric（指标）**：可挂载在任何层级上。包含指标名称、当前值、趋势、所属层级（全局/机器/部门/任务/Agent）。

这五个实体之间的关系定义清楚了，前端怎么画都行。今天画成终端墙，明天画成看板，后天画成 3D 大楼，数据模型都不用变。

## 九、设计原则

### 9.1 三维导航，二维操作

3D 场景用于全局导航和空间感知。但用户真正阅读信息和操作的界面自然过渡为 2D。类似 Google Earth 从太空缩放到街景的体验——你在 3D 空间里移动，但在每个层级上看到的信息都是平面化的、可读的。

### 9.2 聚合 + 异常检测

默认状态是平静的仪表盘，几个大数字，一切绿色。正常运行的 Agent 不需要用户关注。出问题时相关区域变红/闪烁，自动弹出详情。想深入看时点进去逐层下钻。想炫的时候切到终端墙视图，几十个窗口同时滚动。

### 9.3 拒绝过度工程化

每个阶段的交付物都是可用的产品，而不是半成品。保持作为海上小船随 AI 水平面上涨的灵活性。技术架构的重心放在数据模型和数据管道上，前端展示层可以随 AI 能力进化随时替换。

### 9.4 简单到傻子都能懂

控制复杂度，避免用户认知负荷过载。用户不需要配置模板，不需要理解四维坐标，只需要看到大楼、点进去、看到看板、点进去、看到终端。所有复杂性通过层级聚合自然消化。

### 9.5 视觉震撼力

截图或录屏发到社交媒体就能产生流量。深色科技风格背景、粒子特效、光线流动、终端字符滚动——这些视觉元素既服务于信息传达，也服务于传播。

## 十、实施阶段

### 第一阶段：终端墙（1-2 周）

一个网页，上面铺满实时滚动的 Agent 终端窗口。连接本地机器上正在运行的 OpenClaw Session 和 Claude Code Session，把它们的输出流实时展示在网格化的终端窗口里。每个窗口上方标注 Session 名称和运行时长。

**不做**：看板、部门、任务状态、指标、3D、交互输入、云服务器连接。

**意义**：解决最关键的技术问题——实时数据管道。同时，50 个终端同时滚动的截图就是炸裂的内容素材。

### 第二阶段：指标层 + 基础数据模型（2-3 周）

顶部加指标条：当前运行 Agent 数、今日完成任务数、今日 Token 消耗、运行总时长。后端建立五个核心实体的数据模型（Machine、Department、Task、Session、Metric），建立 Agent Session 与 Task 之间的关系。

### 第三阶段：看板视图 + 部门（3-4 周）

两个可切换的视图：终端墙视图和看板视图。看板视图中，左侧是部门列表，点击某个部门进入三列看板。任务卡片可展开查看 Agent 终端输出。用户通过与 OpenClaw 对话自动创建任务并分配到对应部门。

### 第四阶段：3D 大楼 + 多机器（4-5 周）

引入 3D 场景，实现大楼隐喻。建筑 → 楼层 → 看板 → 终端的缩放层级和过渡动画。支持跨机器连接：云服务器上的 Agent 也接入同一个看板。全局仪表盘作为默认首页。

### 第五阶段：动画与视觉打磨（2-3 周）

任务流转动画（粒子特效、光线轨迹）、Agent 活动指示动画、新建筑浮现动画、数字计数器滚动效果、深色科技感视觉风格。确保截图和录屏的传播力。

### 第六阶段及以后：根据实际使用迭代

专用视图（客服/内容/实验）、手势控制、Leader Agent 自动配置指标、集群合并动画、终端内交互输入、多租户与权限等。具体方向根据实际使用中的最大痛点和 AI 能力的进化来决定。

## 十一、核心矛盾与解法

**矛盾**：几万个 Agent 需要展示，但人的注意力只能处理 7±2 个信息块。

**解法**：不是试图同时展示所有东西，而是聚合 + 异常检测 + 层级下钻。正常运行的 Agent 只需要一个绿色状态灯。用户真正需要看到的是：哪个 Agent 挂了、哪个任务卡住了、哪个指标异常了。

最终形态：

- **日常**：一面平静的仪表盘，几个大数字，一切绿色
- **出问题时**：相关区域变红/闪烁，自动弹出详情
- **想深入看时**：点进去逐层下钻
- **想炫的时候**：手掌张开，切到终端墙平铺模式，几十个窗口同时滚动

这样同时满足了"日常好用"和"截图震撼"两个需求。
